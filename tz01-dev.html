01

Захоплення GCLID

Frontend ~1 день
Зберегти gclid при вході на сайт
Problem
Коли user клікає рекламу в Google, в URL з'являється gclid=CjwK.... Якщо ми його не збережемо одразу — він зникне при першому переході. Без нього не можна атрибутувати конверсію.
Solve
При завантаженні кожної сторінки читаємо gclid з URL. Якщо є — пишемо в localStorage і cookie (TTL 90 днів). Cookie — fallback для Safari ITP.
Приклад · front-end snippet
// gclid-tracker.js — підключити на кожній сторінці
const params = new URLSearchParams(window.location.search);
const gclid = params.get('gclid');

if (gclid) {
  localStorage.setItem('gclid_data', JSON.stringify({
    gclid: gclid,
    captured_at: Date.now(),
    landing: window.location.pathname
  }));
  // cookie — fallback Safari / iOS
  document.cookie = `gclid=${gclid}; max-age=${90*86400}; path=/; SameSite=Lax`;
}
Tips
  • Не перезаписуй gclid якщо він вже є і не старіший 90 днів — перший клік завжди пріоритетніший.
  • Разом з gclid зберігай utm_campaign, utm_medium — знадобляться для зведення аналітики.
  • Перевір що скрипт є на всіх лендінгах, не тільки на головній.
Acceptance criteria
  • При кліку з Google Ads gclid зберігається в localStorage і cookie.
  • Якщо gclid в URL немає — нічого не перезаписується.
  • TTL cookie = 90 днів.
  • Разом з gclid зберігаються utm_campaign, utm_medium, landing_page.
02

Трекінг кліку на партнера

Backend ~1–2 дні
Генерувати click_id і передавати його партнеру як subid
Problem
Коли user клікає "Visit Site" — він йде на партнера. Ми не знаємо хто саме перейшов і не можемо зматчити майбутній постбек з нашим gclid.
Solve
Перед редиректом генеруємо click_id, зберігаємо пару click_id ↔ gclid в БД, і передаємо click_id партнеру через параметр subid у посиланні.
Приклад · POST /api/click/track
# Request
POST /api/click/track
{
  "session_id": "sess_abc",
  "partner_id": "japandates",
  "offer_id":  "asian_main",
  "destination_url": "https://japandates.com"
}

# Server logic
click_id = uuid()
gclid    = db.sessions.get(session_id).gclid

db.clicks.insert({
  click_id, gclid, partner_id, offer_id, clicked_at: now()
})

# Response — фронт редиректить на цей URL
{ "redirect_url": "https://japandates.com?subid={click_id}" }
Tips
  • Назва параметра залежить від партнера: може бути subid, s1, aff_sub — уточни у кожного партнера.
  • Зберігай offer_id і partner_id окремо — знадобиться для аналітики яка позиція конвертує краще.
  • Redis для lookup session_id → gclid — швидше ніж йти в PostgreSQL на кожен клік.
Acceptance criteria
  • Кожен клік на партнерський лінк створює запис в таблиці clicks.
  • В записі є click_id, gclid, partner_id, offer_id, clicked_at.
  • Redirect URL містить click_id як subid параметр.
  • Якщо gclid не знайдено — клік все одно фіксується, gclid = null.
03

Обробка постбека від партнера

Backend ~1–2 дні
Прийняти постбек, зматчити з gclid, поставити в чергу
Problem
Партнер надсилає POST/GET на наш endpoint коли user конвертує. Нам треба прийняти це, знайти gclid по click_id і не загубити якщо Google Ads API тимчасово недоступний.
Solve
Endpoint приймає постбек, одразу відповідає 200 OK партнеру, асинхронно кладе задачу в чергу. Worker окремо підбирає і відправляє в Google Ads.
Приклад · GET /postback
# Партнер надсилає
GET /postback?click_id=clk_abc123&type=registration&value=0

# Server logic
click   = db.clicks.get(click_id="clk_abc123")
gclid   = click.gclid  # може бути None якщо не зберегли

db.conversions.insert({
  click_id, gclid,
  type: "registration",
  value: 0, status: "pending"
})

# Кладемо в чергу (SQS / RabbitMQ)
queue.send({ "task": "send_to_gads", "click_id": "clk_abc123" })

return "OK", 200  # відповідаємо партнеру одразу
Tips
  • Відповідай партнеру 200 OK одразу — не чекай поки відправимо в Google Ads. Якщо відповідь затримається, партнер буде слати повторно.
  • Зберігай сирий постбек цілком (raw_payload) — знадобиться для дебагу.
  • Перевіряй click_id на існування перед записом — якщо click не знайдено, лог + ігнорувати (не падати з 500).
Acceptance criteria
  • Endpoint відповідає 200 OK протягом 200ms незалежно від стану Google Ads API.
  • Конверсія зберігається в conversions зі статусом pending.
  • Задача потрапляє в чергу одразу після запису.
  • Якщо click_id не знайдено — повертаємо 200, пишемо в лог, не падаємо.
04

Відправка конверсії в Google Ads

Backend · Worker ~2 дні
Worker читає чергу і відправляє в Google Ads Conversions API
Problem
Google Ads оптимізує покази тільки якщо бачить конверсії. Без передачі — алгоритм сліпий, ми платимо за кліки які не конвертують.
Solve
Worker підбирає задачі з черги, по click_id знаходить gclid, формує запит до ConversionUploadService і відправляє. Дедуплікація через order_id.
Приклад · Python · google-ads-python
from google.ads.googleads.client import GoogleAdsClient

client = GoogleAdsClient.load_from_env()

click_conversion = client.get_type('ClickConversion')
click_conversion.gclid                 = gclid
click_conversion.conversion_action     = "customers/{CID}/conversionActions/{AID}"
click_conversion.conversion_date_time  = "2025-04-01 14:00:00+02:00"
click_conversion.conversion_value      = value   # 0 для registration
click_conversion.order_id              = click_id # дедуплікація!

service = client.get_service('ConversionUploadService')
service.upload_click_conversions(
  customer_id=CUSTOMER_ID,
  conversions=[click_conversion],
  partial_failure=True
)

# Docs: developers.google.com/google-ads/api/docs/conversion-tracking/upload-clicks
Tips
  • order_id = click_id — Google Ads ігнорує дублікати по order_id. Без цього одна конверсія може порахуватись двічі.
  • Якщо gclid = null — спробуй Enhanced Conversions: передай SHA-256 хеш email користувача як user_identifier. Google сам заматчить. Docs →
  • Постбек може прийти через 30 днів — API підтримує backdated conversions до 90 днів. Просто передай реальну дату конверсії.
  • Встав partial_failure=True — інакше одна помилка вб'є весь батч.
Acceptance criteria
  • Worker обробляє задачу з черги і відправляє конверсію в Google Ads.
  • Після успіху статус конверсії в БД змінюється на sent.
  • При помилці — статус failed, retry через 5 хв (max 3 спроби).
  • Дублікат постбека не створює дублікат конверсії в Google Ads (order_id перевірено).
  • Якщо gclid = null — спроба Enhanced Conversions, лог результату.
05

ETL: збір статистики з Google Ads

ETL · Флоу B ~1–2 дні
Щодня тягнути витрати з кабінету в data storage
Problem
Ми знаємо revenue (з постбеків), але не знаємо spend. Без витрат неможливо порахувати ROAS, CAC і вирішити чи прибуткова закупівля взагалі.
Solve
Щоденний cron-job тягне статистику по кампаніях/ключах через GAQL і зберігає в таблицю google_ads_stats. Джойн з clicks по campaign_id + date дає повну картину.
Приклад · GAQL запит
-- Запускати щодня, date = вчора
SELECT
  campaign.id,
  campaign.name,
  ad_group.id,
  segments.keyword.info.text,
  segments.date,
  metrics.impressions,
  metrics.clicks,
  metrics.cost_micros        -- divide by 1_000_000 → USD
FROM keyword_view
WHERE segments.date = '{yesterday}'
  AND campaign.status = 'ENABLED'

-- Docs: developers.google.com/google-ads/api/docs/query/overview
Tips
  • Зберігай campaign_id + date як composite key — це ключ для джойну з даними конверсій.
  • Дані в Google Ads затримуються на ~3 години. Запускай ETL не раніше ніж о 6:00 ранку для вчорашнього дня.
  • Якщо є кілька Google Ads акаунтів — використовуй MCC (Manager Account) і перебирай customer_ids в циклі.
Acceptance criteria
  • ETL запускається щодня о 06:00 UTC і тягне дані за вчора.
  • Таблиця google_ads_stats містить: campaign_id, ad_group_id, keyword, date, impressions, clicks, cost_usd.
  • Повторний запуск за той самий день не дублює дані (upsert по campaign_id + date).
  • Джойн google_ads_stats.campaign_id + date з таблицею конверсій повертає коректний ROAS.

Мінімальна схема БД

PostgreSQL 
sessions    (session_id PK, gclid, utm_campaign, landing_page, created_at)
clicks      (click_id PK,   session_id FK, gclid, partner_id, offer_id, clicked_at)
conversions (conv_id PK,    click_id FK,  gclid, type, value, status, received_at, sent_at)

google_ads_stats (campaign_id, ad_group_id, keyword, date,
                  impressions, clicks, cost_usd, fetched_at)
-- PK: campaign_id + date (upsert)
